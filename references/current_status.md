# Current Codebase Status

Last updated: 2026-05-21

## Runtime stack

The current executable path is:

```text
ESP-ROM -> ESP-IDF v5.5.3 2nd stage bootloader -> Rust no_std app
```

The Rust app now uses:

- `riscv-rt = 0.17.1`
- custom PAC from `pac/`
- direct low-level register access
- manually defined ESP-IDF app descriptor in `src/esp_app_desc.rs`

`esp-riscv-rt` is no longer used.

## Active linker setup

Active linker arguments are only:

```toml
-Tmemory.x
-Tlink.x
```

`link.x` is generated by `riscv-rt`. The project-owned linker file is `memory.x`.

Current `memory.x` is intentionally minimal:

```ld
MEMORY
{
  RAM : ORIGIN = 0x4ff00000, LENGTH = 64K
  FLASH : ORIGIN = 0x40000020, LENGTH = 1M
}
```

It also inserts a small `.text_gap` before `.rodata` so `espflash` emits separate mapped segments for text and rodata-like data.

## Active flash command

`cargo run` uses the runner in `.cargo/config.toml`:

```bash
espflash flash --monitor \
  --chip esp32p4 \
  --flash-size 16mb \
  --flash-mode qio \
  --flash-freq 80mhz \
  --partition-table partitions.csv
```

The app is flashed to the `factory` partition at `0x10000`.

## Current boot result

The ESP-IDF bootloader now reaches app loading without the earlier blocking failures.

Resolved symptoms:

- app descriptor/eFuse block revision rejection
- `rom_index == 2` bootloader assert
- immediate instruction access fault from IROM entry experiment

Current observed boot log reaches:

```text
I boot: Loaded app from partition at offset 0x10000
I boot: Disabling RNG early entropy source...
```

After that, there is no confirmed app UART output. The chip appears to reset after some time.

This means the current investigation has moved from boot image compatibility to app runtime behavior.

## Current ESP image shape

With the current simple linker setup, `espflash save-image` emits multiple segments, including dummy/padding segments with `vaddr=0`.

Observed shape:

```text
segments = 4
entry = 0x40000020

segment 0: vaddr=0x00000000 size=0x05ec
segment 1: vaddr=0x40000614 size=0x029c map
segment 2: vaddr=0x00000000 size=0xf760
segment 3: vaddr=0x40000020 size=0x05d0 map
```

Even though this is not clean, the ESP-IDF bootloader currently accepts it and transitions to the app.

## Current output path

The app writes directly to UART0 FIFO:

```rust
uart0.fifo().write(|w| unsafe { w.bits(byte as u32) });
```

No UART clock, baudrate, or pin-mux initialization is done by the app yet. The monitor is connected to `/dev/ttyACM0`, which may be USB Serial/JTAG CDC rather than UART0 pins.

See known issues for details.
