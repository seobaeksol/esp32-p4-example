# Linker Script Notes for Current Codebase

Last updated: 2026-05-21

This is a practical guide for the current linker setup. For historical linker experiments, see the other analysis documents under `references/`.

## Active linker scripts


| Script | Source | Purpose |
|---|---|---|
| `memory.x` | project file | memory regions, aliases, current `.text_gap` tweak |

## Current `memory.x`

```ld
MEMORY
{
  RAM : ORIGIN = 0x4ff00000, LENGTH = 64K
  FLASH : ORIGIN = 0x40000020, LENGTH = 1M
}
```

Aliases used by `riscv-rt`:

```ld
REGION_ALIAS("REGION_TEXT", FLASH);
REGION_ALIAS("REGION_RODATA", FLASH);
REGION_ALIAS("REGION_DATA", RAM);
REGION_ALIAS("REGION_BSS", RAM);
REGION_ALIAS("REGION_HEAP", RAM);
REGION_ALIAS("REGION_STACK", RAM);
```

## `MEMORY`

`MEMORY` defines address ranges available to the linker.

```ld
NAME : ORIGIN = address, LENGTH = size
```

Examples:

```ld
FLASH : ORIGIN = 0x40000020, LENGTH = 1M
RAM   : ORIGIN = 0x4ff00000, LENGTH = 64K
```

`ORIGIN(NAME)` and `LENGTH(NAME)` can be used in expressions.

## `REGION_ALIAS`

`riscv-rt` does not know the project-specific region names. It expects aliases such as `REGION_TEXT` and `REGION_STACK`.

```ld
REGION_ALIAS("REGION_TEXT", FLASH);
```

This means `riscv-rt`'s `.text` output section is placed in `FLASH`.

## `SECTIONS`

A linker script can define or insert output sections:

```ld
SECTIONS {
  .text_gap (NOLOAD): {
    . = . + 4;
    . = ALIGN(4) + 0x20;
  } > FLASH
}
INSERT BEFORE .rodata;
```

The current `.text_gap` is inserted before `riscv-rt`'s `.rodata` output section.

## Location counter `.`

`.` is the current address inside the output section.

```ld
. = . + 4;
. = ALIGN(4) + 0x20;
```

This advances the section and creates a gap. `NOLOAD` means the section occupies address space but does not contribute initialized bytes directly.

## `INSERT BEFORE` / `INSERT AFTER`

When multiple scripts are used, `INSERT BEFORE` can place a new section relative to a section defined by another script.

```ld
INSERT BEFORE .rodata;
```

This is useful because `link.x` is generated by `riscv-rt`, while this project still needs small ESP-specific placement tweaks.

## `KEEP`

`KEEP` prevents `--gc-sections` from removing an input section.

Example from app descriptor experiments:

```ld
KEEP(*(.flash.appdesc));
```

The current code relies on `#[used]` and `espflash` descriptor discovery, but if a custom appdesc output section is reintroduced, use `KEEP`.

## VMA, file offset, ESP image segment

Important concepts:

| Concept | Meaning |
|---|---|
| VMA | runtime/load address shown by `readelf -S` Address |
| ELF file offset | location inside the ELF file |
| ESP image segment | segment emitted by `espflash save-image` |
| partition paddr | ESP image offset plus app partition offset |

Useful commands:

```bash
readelf -S target/riscv32imafc-unknown-none-elf/release/esp32-p4-example
readelf -l target/riscv32imafc-unknown-none-elf/release/esp32-p4-example
```

ESP image inspection:

```bash
espflash save-image \
  --chip esp32p4 \
  --flash-size 16mb \
  --flash-mode qio \
  --flash-freq 80mhz \
  target/riscv32imafc-unknown-none-elf/release/esp32-p4-example \
  /tmp/esp32-p4-example.bin
```

```bash
python3 - <<'PY'
from pathlib import Path
import struct
b = Path('/tmp/esp32-p4-example.bin').read_bytes()
print('segments', b[1], 'entry', hex(struct.unpack_from('<I', b, 4)[0]))
o = 24
for i in range(b[1]):
    addr, size = struct.unpack_from('<II', b, o)
    o += 8
    print(i, hex(addr), hex(size), 'data_off', hex(o))
    o += size
PY
```

## Current known linker/image risk

The current image is accepted by the bootloader, but it is not canonical and includes dummy `vaddr=0` segments. See:

- `references/known_issues/ki-005-noncanonical-esp-image-segments.md`

Do not optimize the image layout further until runtime visibility/reset issues are understood.
